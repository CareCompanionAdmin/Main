# 2026-05-22 — Pro QA admin section

## Summary
New `/admin/pro-qa/*` super-admin-only workspace with four tabs (Intro,
Info, Requested Checks, Issue Tracker). Data lives on the shared support
DB (prod RDS) so dev and prod see the same rows. Attachments use a new
S3 prefix `pro-qa/` under the existing bucket.

## Pre-deploy: prod RDS schema + role grants

Because dev routes `pro_qa_*` reads/writes through `SUPPORT_DB_DSN` (= prod
RDS), the tables must exist on prod RDS **before** dev can even be
smoke-tested — and **before** the prod code deploy.

Apply this once against prod RDS (requires `claude-superadmin` temp creds
per `reference_prod_db_access.md`):

```bash
PGPASSWORD="$PROD_DB_PASSWORD" psql \
  -h carecompanion-db.cns7qg5iujxu.us-east-1.rds.amazonaws.com \
  -U carecompanion -d carecompanion \
  -f migrations/00039_pro_qa.sql

PGPASSWORD="$PROD_DB_PASSWORD" psql \
  -h carecompanion-db.cns7qg5iujxu.us-east-1.rds.amazonaws.com \
  -U carecompanion -d carecompanion <<'SQL'
GRANT SELECT, INSERT, UPDATE, DELETE ON
  pro_qa_info, pro_qa_requested_checks, pro_qa_issues,
  pro_qa_issue_comments, pro_qa_issue_attachments
TO carecomp_support_dev;
GRANT USAGE, SELECT ON SEQUENCE pro_qa_issues_issue_number_seq TO carecomp_support_dev;
SQL
```

Verify the grant:
```sql
SET ROLE carecomp_support_dev;
SELECT count(*) FROM pro_qa_info;  -- expect 1
RESET ROLE;
```

## Rollback
- Code: revert the feature commits and redeploy.
- Schema: tables are isolated; safe to leave in place. If hard cleanup
  is required, run the rollback DROP block at the top of
  `migrations/00039_pro_qa.sql` against prod RDS.

bucket. No new bucket or IAM change required — the existing app role
