Fix: Medication schedule timezone and display issues

Problems resolved:
- Day rollover bug: Child dashboard used server UTC time instead of user's timezone, causing medications to show on the wrong day (e.g., Monday meds appearing Sunday evening for US timezones)
- Quick Summary wrong day: Daily/weekly/monthly Quick Summary computed "today" in UTC, showing tomorrow's date for US evening users
- Daily logs navigation: Previous/next day buttons skipped days because date strings were parsed as UTC midnight, shifting the day backward when converted to local time
- All log endpoints (behavior, diet, sleep, bowel, weight, speech, sensory, social, therapy, seizure, health events) used UTC for default date ranges instead of user timezone
- API timezone mismatch: Medication "due today" and log endpoints parsed dates in UTC, not the user's configured timezone
- Log date shift: Logging a medication late at night could record it under the next day due to UTC conversion
- Time display: Scheduled times showed raw database format (e.g., "08:00:00") instead of user-friendly format (e.g., "8:00 AM")
- Time format preference: formatTimeStr now respects user's 12h/24h setting (was always showing 12-hour)
- Time of day capitalization: "morning", "with_breakfast" etc. now display as "Morning", "With Breakfast"
- Time of day mismatch: Schedule creation/edit forms now auto-update time_of_day (morning/afternoon/evening/night) when the user changes the scheduled time, preventing mismatches like "morning (4:00 PM)"
- Dev admin: Added public access toggle for dev environment (DevTesting security group)
- Version log: Added change summary display for uncommitted changes
